## 1. 결론

제안한 순서인 **Next.js 설치 → 기본 폴더 구성 → Figma Variables를 디자인 토큰으로 변환 → 대표 화면 UI 제작 → 기능 개발**은 큰 방향이 맞다. 다만 UI가 완성된 뒤에만 개발을 시작하면 6일 일정에서는 통합 위험이 너무 크다.

따라서 다음처럼 겹쳐 진행한다.

1. 첫날에 Next.js 골격, 토큰, 라우트 셸을 만든다.
2. 대표 화면 UI를 만드는 동안 타입, 상태 흐름, IndexedDB 저장소를 병렬로 구현한다.
3. 4일차까지 목업 데이터로 전체 수직 흐름을 연결한다.
4. 5일차에 실제 AI 연동을 붙이고 6일차에 MVP를 동결한다.
5. 7~9일차는 음성·사진·잠금·PWA 캐시와 품질 보강에 사용한다.

6일은 핵심 데모를 만드는 최단 일정이고, 9일은 실패 대응과 접근성·모바일 품질을 확보하는 현실적인 일정이다.

## 2. 프로토타입 완료 기준

393px 폭의 모바일 브라우저를 기준으로 다음 흐름이 끊기지 않아야 한다.

1. 사용자가 프로필과 기본 의료정보를 입력한다.
2. 텍스트 또는 빠른 선택지로 면담 질문에 답한다.
3. 앱이 다음 질문을 생성하거나 수동 질문 입력으로 대체한다.
4. 면담 종료 시 S/O 구조의 요약을 생성한다.
5. 생성 실패 시 사용자가 직접 요약을 작성할 수 있다.
6. 저장된 기록을 다시 열 수 있다.

MVP는 회원가입, 서버 데이터베이스, 완전한 오프라인 동작을 요구하지 않는다. 사용자 데이터는 IndexedDB에 저장하며 AI 호출은 Next.js Route Handler를 통해 수행한다.

## 4. 구현 원칙

### 4.1 대표 화면 선정

대표 화면은 홈이 아니라 `/interview/new`로 한다. 이 화면에서 디자인 시스템과 핵심 상호작용을 동시에 검증할 수 있기 때문이다.

대표 화면에는 최소한 다음 상태가 포함되어야 한다.

- 기본 질문과 텍스트 입력
- 빠른 선택지
- 전송 가능·불가능 상태
- 응답 생성 중
- AI 오류와 수동 입력 전환
- 음성 듣기·읽기 상태
- 이미지 첨부 상태
- 이전 답변 확인 또는 정정 진입

### 4.2 디자인 토큰

Figma Variables는 그대로 복사하는 것이 아니라 다음 세 층으로 변환한다.

- Primitive: 원시 색상, 크기, 반경, 간격
- Semantic: 배경, 표면, 본문, 보조 텍스트, 위험, 포커스
- Component: 버튼, 입력창, 카드처럼 제한적으로 필요한 별칭

기본 모드는 `:root`의 CSS Custom Properties로 만들고, Figma에 다크 모드나 고대비 모드가 있으면 별도 선택자에 보존한다. 변수 이름은 시각적 값이 아니라 역할을 표현해야 한다.

### 4.3 저장소와 캐시

- IndexedDB: 프로필, 의료정보, 면담, 답변, 요약 같은 사용자 데이터
- Cache Storage: 앱 셸, 폰트, 아이콘 등 재생성 가능한 정적 자산
- 저장 성공은 IndexedDB 요청 성공이 아니라 트랜잭션 완료까지 확인한다.
- 서비스 워커는 데이터 스키마와 주요 흐름이 안정된 뒤 도입한다.

### 4.4 AI 통합

UI는 먼저 결정론적 목업 응답으로 연결한다. 실제 모델 호출은 Route Handler 뒤에 배치하고 요청·응답 스키마를 검증한다. 모델 장애, 시간 초과, 잘못된 JSON은 모두 수동 질문 또는 수동 요약으로 복구되어야 한다.

## 6. TTS 구현 방안

현재 Next.js PWA에서는 브라우저의 `window.speechSynthesis`를 사용한다.

- Client Component의 명시적 “읽어주기” 버튼에서만 실행한다.
- 기본 자동 읽기는 개인정보 보호와 주변 노출 위험 때문에 끈다.
- 질문, 안전 안내, 최종 요약만 읽기 대상으로 제한한다.
- 가능한 경우 `ko-KR` 음성을 선택하되 음성 목록이 늦게 로드되는 상황을 처리한다.
- 새 발화 전 기존 발화를 취소하고, 화면 이동·컴포넌트 해제 시에도 중단한다.
- 미지원 또는 오류 시 텍스트 표시가 항상 남아 있어야 한다.

iOS 네이티브로 전환할 경우에는 `AVSpeechSynthesizer`가 더 일관된 선택이다. 그러나 SwiftUI 앱, 데이터 계층, API 클라이언트, 저장소를 별도로 구현해야 하므로 현재 6~9일 PWA 범위에는 포함하지 않는다.

## 7. 역할 분담

### 사용자

- Figma Variables와 화면 상태의 최종 시각 결정
- 대표 화면 UI 제작과 디자인 검수
- 6일차 MVP 범위 변경 승인

### 개발

- Next.js 골격과 폴더 구성
- 타입, 상태 흐름, IndexedDB 저장소
- API Route Handler와 AI 스키마 검증
- 로딩·오류·대체 경로
- 테스트, 접근성, PWA 통합

대표 UI와 개발 기반은 2일차부터 병렬로 진행하며, UI 인계 완료를 기다린 뒤 기능 개발을 시작하지 않는다.

## 8. 위험과 중단 규칙

| 위험 | 대응 | 중단 기준 |
|---|---|---|

| MedGemma 접근·응답 불안정 | 목업 제공자와 수동 대체 유지 | 5일차 종료까지 해결되지 않으면 데모는 대체 경로 사용 |
| 음성 API의 iOS 브라우저 편차 | 텍스트 입력·읽기 버튼을 기본 경로로 유지 | 핵심 면담 흐름을 음성 기능에 의존시키지 않음 |
| 범위 증가 | 6일차 필수 목록과 교환되는 항목 명시 | 6일차 이후 신규 기능 추가 금지 |

## 9. 검증과 완료 증거

각 날짜의 실제 명령, 테스트 결과, 변경 파일, 장애와 결정은 날짜별 작업일지에 기록한다. 패키지 도구와 테스트 스택은 1일차에 저장소 환경을 확인한 뒤 확정하며, 확인 전에는 명령을 문서에 가정해서 적지 않는다.

최종 완료를 주장하려면 다음 증거가 있어야 한다.

- 프로덕션 빌드 성공 기록
- 핵심 흐름의 자동 또는 반복 가능한 수동 테스트 결과
- 393px 모바일 화면 확인
- 새로고침·재실행 후 IndexedDB 복원 확인
- AI 성공, 시간 초과, 오류, 수동 대체 시나리오 확인
- 키보드 탐색과 주요 스크린리더 레이블 확인
- 실제 모바일 기기에서 데모 리허설 완료
