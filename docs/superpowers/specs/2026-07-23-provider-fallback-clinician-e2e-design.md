# Provider 장애 복구·의료진 경로 E2E 설계

- 작성일: 2026-07-23
- 상태: 사용자 설계 승인 완료
- 대상: 공개 사용자 온보딩부터 AI 문진 복구, 완료 기록, 의료진용 화면까지

## 배경

현재 AI 문진 서비스는 질문 provider 호출이 실패하면 같은 AI 문진 안에 결정론적 대체 질문을 저장한다. 요약 provider 호출이 실패하거나 유효한 요약을 반환하지 않으면 사용자가 입력한 답변을 근거로 결정론적 요약을 저장한다.

단위 테스트는 두 복구 동작을 검증하지만, 실제 공개 화면과 IndexedDB를 거쳐 완료 기록과 의료진용 화면까지 이어지는 경로는 아직 하나의 E2E 테스트로 증명되지 않았다. 데모에서 provider 장애가 발생해도 사용자가 문진을 끝내고 의료진에게 기록을 보여줄 수 있다는 근거가 필요하다.

## 결정

기존 런타임 동작과 저장 스키마는 변경하지 않는다. Playwright에서 질문·요약 API의 실제 HTTP 실패 응답을 모의하고, 공개 사용자 경로 전체가 기존 결정론적 복구 로직으로 완료되는지 검증한다.

복구된 기록의 `mode`는 `ai`로 유지한다. 기록 목록·상세·의료진용 화면에는 `AI 문진`으로 표시하고, 요약 출처는 기존 `source: "manual"`을 사용해 `입력 내용 정리`로 표시한다.

## 검토한 대안

### 별도 수동 문진 화면으로 이동

장애 전환을 사용자에게 더 명시적으로 보여줄 수 있다. 그러나 진행 중인 AI 문진과 수동 문진 사이의 답변 병합, 복원, 중복 기록 방지 규칙이 새로 필요하다. 빠른 데모 검증 범위를 넘으므로 채택하지 않는다.

### 장애·복구 출처를 위한 DB 필드 추가

운영 추적성은 높아진다. 반면 스키마 마이그레이션, view model, 모든 기록 화면의 변경이 필요하다. 현재 UI는 문진 방식과 요약 출처를 이미 구분하므로 이번 범위에는 과도하다.

## 사용자 흐름

1. 사용자가 공개 온보딩에서 AI 전송에 동의한다.
2. 홈에서 `AI 문진 시작하기`를 선택한다.
3. 첫 결정론적 질문에 합성 답변을 입력한다.
4. `/api/ai/question`이 HTTP 503을 반환한다.
5. 저장된 답변과 겹치지 않는 결정론적 대체 질문이 화면에 표시된다.
6. 사용자가 대체 질문에 답한다.
7. `/api/ai/summary`가 HTTP 503을 반환한다.
8. 입력 답변 기반 요약과 복구 안내가 검토 화면에 표시된다.
9. 사용자가 문진을 완료하고 홈으로 이동한다.
10. 기록 목록에서 `완료 · AI 문진` 항목을 연다.
11. 기록 상세에서 `입력 내용 정리`와 원문 답변을 확인한다.
12. 의료진용 화면에서 같은 유형·요약 출처·원문 답변을 확인한다.

## 구현 범위

### E2E 테스트

`tests/e2e/public-ai-interview.spec.ts`에 독립 시나리오를 추가한다.

- 온보딩은 기존 합성 헬퍼를 재사용한다.
- 질문과 요약 route는 각각 HTTP 503을 반환한다.
- 대체 질문의 고정 문구와 입력 컨트롤을 통해 실제 복구 화면을 확인한다.
- 완료 뒤 IndexedDB에서 해당 완료 문진 ID를 읽는다.
- 기록 목록, 상세, 의료진용 경로를 실제 링크로 이동한다.
- 각 화면에서 `AI 문진`, `입력 내용 정리`, 최초 답변과 대체 질문 답변을 확인한다.
- 외부 AI·Modal·GPU는 호출하지 않는다.

필요할 경우 완료 문진 ID 조회 헬퍼는 테스트 파일 안에 두어 제품 코드에 테스트 전용 API를 추가하지 않는다.

### 문서

검증 체크리스트의 `Task 1B provider 실패 manual 문진·기록 확인` 항목을 실제 완료 증거와 일치하도록 갱신한다. 표현은 구현 의미에 맞게 “AI 문진 내 결정론적 대체 질문·입력 기반 요약 복구”임을 명확히 한다.

## 오류 및 안전 경계

- 503 응답은 앱 서버가 아닌 Playwright route에서 생성해 테스트를 결정론적으로 유지한다.
- 실제 개인정보 대신 합성 이름·생년월일·증상만 사용한다.
- API 요청 payload의 개인정보 allowlist 검증은 기존 정상 경로 테스트가 계속 담당한다.
- 기존 긴급 안전 중단, 동의 철회, 전체 삭제의 stale 응답 방어는 변경하지 않는다.
- `.gitignore`의 사용자 변경은 이 작업의 커밋에 포함하지 않는다.

## 검증 전략

TDD 순서를 따른다.

1. 새 E2E 시나리오를 작성하고 대상 테스트만 실행해 RED를 확인한다.
2. 실패 원인이 테스트 헬퍼 또는 기존 화면 증거 부족이면 최소 범위로 보완한다.
3. 대상 Chromium E2E를 GREEN으로 만든다.
4. 문서 정합성과 `git diff --check`를 확인한다.
5. 커밋·푸시 전 통합 지점에서 전체 E2E를 한 번 실행한다. `npm run test:e2e`가 production build를 포함하므로 별도 build는 반복하지 않는다.

## 완료 조건

- HTTP 503 질문 실패 뒤 결정론적 대체 질문으로 계속 진행한다.
- HTTP 503 요약 실패 뒤 입력 답변 기반 요약으로 완료한다.
- 완료 기록은 목록·상세·의료진용 화면에서 모두 열린다.
- 기록 유형은 `AI 문진`, 요약 출처는 `입력 내용 정리`로 표시된다.
- 실제 외부 AI 호출 없이 경로가 재현 가능하다.
- 체크리스트와 실행 증거가 현재 코드 및 테스트 결과와 일치한다.
