## 완료 작업

- Figma `Prototype` 파일의 `prototype` 페이지와 `최종 prototype` 섹션을 읽기 전용으로 조사했다.
- 섹션의 직접 자식 제품 프레임 46개를 시작·홈·신규 문진·기록·프로필 영역으로 분류했다.
- 각 필수 UI의 Figma 이름과 node ID를 문서화했다.
- 입력값·키보드·선택·첨부 상태를 별도 라우트가 아닌 같은 화면의 상태 변형으로 구분했다.
- 설명용 프레임, 주석 텍스트, 참조 이미지, 섹션 밖 과거 시안을 구현 범위에서 제외했다.
- `/interview/new`의 전체 대화 스크롤, 직전 답변·현재 질문 강조, 명시적 `다음` 계약을 승인했다.
- 선택·텍스트·음성 초안을 함께 보존하고 질문마다 확정 시 한 번 저장한 뒤 AI를 호출하도록 정했다.
- 질문 번호·단계·진행률은 표시하지 않고 비동기 동작 상태만 알리도록 정했다.
- 제공된 투명 iPhone 17 PNG와 웹 iOS 상태바를 사용하는 393×852 고정 프레임을 승인했다.
- 반응형·320px reflow는 제외하고 200% 확대 시 바깥·내부 scroll 접근을 검증하도록 변경했다.
- 서버 전용 flag와 URL query로만 접근하는 상태 fixture 9개를 확정했다.
- 상태 직접 진입과 사용자 행동 기반 결정론적 전환을 함께 제공하도록 정했다.
- 저장 300ms 표시 임계값, 900ms fixture 저장, 1,200ms fixture AI 응답을 확정했다.
- 상태별 쉬운 문구, 주요 행동, focus·live region·busy assertion을 기록했다.
- 대표 문진 화면 구현을 테스트 기반, fixture 계약, 디바이스 셸, 대화 입력, 상태 전환, E2E 검증의 6개 작업으로 나눴다.
- Server Component의 fixture allowlist 판정과 Client Component의 결정론적 command adapter 경계를 구현 계획에 명시했다.
- Vitest·Testing Library의 jsdom 단위 테스트 기반과 Playwright Chromium의 production server E2E 실행 계약을 구성했다.
- 첫 한국어 jsdom 계약 테스트 1건과 lint·typecheck를 통과했다.
- 문진 UI 상태·질문·초안·오류·안전·요약 타입과 합성 fixture 9개를 구현했다.
- 서버 flag와 URL 값을 함께 검사하는 순수 allowlist resolver를 구현했다.
- fixture의 ID·접근성 기대값·금지 필드 계약을 테스트 우선으로 검증했다.
- 사용자 제공 iPhone 17 PNG를 원본 hash 그대로 정적 자산에 복사했다.
- 393×852 고정 viewport, 웹 iOS 상태바, 홈 링크가 있는 62px 문진 헤더를 구현했다.
- 프레임과 상태바를 보조기술에서 제외하고 프레임 overlay의 pointer event를 차단했다.
- 전체 확정 대화와 현재 질문을 분리하고 하나의 내부 scroll log로 구현했다.
- 단일·다중 선택, 자유 텍스트, 음성 입력 모드와 명시적 `다음`을 구현했다.
- 선택값과 수정한 텍스트를 함께 보존하고 같은 질문의 중복 제출을 차단했다.
- Vitest 전역 API 없이도 테스트마다 DOM이 정리되도록 명시적 RTL cleanup을 추가했다.
- `/interview/new` Server Component에서 fixture query를 서버 전용 flag와 allowlist로 제한하고 기본 상태를 연결했다.
- 저장 상태는 300ms 뒤 표시하고, 900ms fixture 저장이 끝난 후에만 1,200ms fixture AI 요청을 시작하도록 구현했다.
- 저장 오류는 기존 초안을 유지한 채 저장부터, AI 오류는 재저장 없이 AI만 재시도하도록 구현했다.
- 주의·긴급 안전 안내와 긴급 행동 후 입력이 복구되지 않는 안전 종료 상태를 구현했다.
- 긴 대화에서 `최신 질문으로 이동`을 제공하고 이동 후 현재 질문 제목으로 초점을 복원했다.
- Vitest가 기존 Node 계약 테스트를 수집하지 않도록 단위 테스트 범위를 `tests/unit`으로 제한했다.
- Playwright에서 앱 viewport 393×852와 전체 프레임 440×900 크기를 측정하고 9개 fixture 직접 진입을 검증했다.
- 긴 대화의 최신 질문 복귀와 Tab·Space·Enter만 사용하는 답변 제출·AI 오류 재시도를 검증했다.
- 1280×1000 Chromium 창에서 전체 기기와 대표 상태 3개의 시각 snapshot을 승인하고 변경 없이 재실행했다.
- 실제 Chrome 창을 열었으나 macOS가 자동 키 입력을 허용하지 않아 200% 확대는 적용하지 못했다. CSS·CDP 확대는 대리 증거로 사용하지 않았다.
- 사용자가 실제 Chrome 200% 확대에서 바깥·내부 scroll, 모든 입력·주요 행동 접근과 visible focus를 확인했다.
