# 문서 관리 규칙

## 목적

모든 문서를 빠짐없이 읽고 갱신할 수 있도록 상위 문서는 탐색과 상태에 집중하고, 상세 내용은 책임별 문서로 분리한다.

## 200줄 제한

- 모든 Markdown 파일은 빈 줄과 front matter를 포함해 최대 200줄이다.
- 상위 문서는 목표, 상태, 권위, 상세 문서 링크만 유지한다.
- 상세 문서는 하나의 기능, 결정, 일정 구간 또는 검증 책임만 다룬다.
- 180줄에 가까워지면 변경 여유를 위해 미리 분리한다.
- 긴 코드·프롬프트·데이터 예시는 별도 상세 문서로 옮기고 링크한다.

200줄은 모델이 읽을 수 있는 절대 한계가 아니라 누락과 수정 충돌을 줄이기 위한 프로젝트 운영 한계다.

## 문서 구조

```text
docs/
├── README.md                 # 전체 문서 허브
├── documentation-policy.md  # 이 규칙
├── plans/
│   ├── <plan>.md             # 활성 상위 계획
│   └── <plan>/               # 계획 상세 문서
├── superpowers/
│   ├── plans/                # 과거 계획
│   └── specs/                # 버전별 설계와 상세 문서
└── worklogs/                 # 날짜별 사실 기록
```

상위 파일과 상세 폴더는 확장자를 제외하고 같은 이름을 사용한다.

## 작성 원칙

1. 같은 요구사항을 여러 문서에 복사하지 않고 기준 문서를 상대 링크로 참조한다.
2. 요구사항과 구현 단위는 기존 `R`, `F`, `AE`, `KTD`, `U` 식별자를 유지한다.
3. 상위 문서에는 현재 상태와 다음 행동을 표시한다.
4. 완료 체크는 구현과 검증 증거가 모두 있을 때만 갱신한다.
5. 날짜가 오래됐다는 이유만으로 문서를 삭제하지 않는다. 대신 활성·과거 상태를 명시한다.
6. 현재 문서와 과거 문서가 충돌하면 [문서 허브](./README.md)의 권위 순서를 따른다.

## 상세 문서 분리 기준

- 제품 목표·사용자 흐름·범위·기술 설계·일정·검증은 서로 다른 파일로 둔다.
- 구현 체크리스트는 날짜 또는 구현 단위별로 나눈다.
- 큰 과거 문서는 원래 순서를 보존한 여러 조각으로 나누고 상위 인덱스에서 연결한다.
- 한 문단이나 코드 블록을 파일 경계에서 자르지 않는다.

## 변경 절차

1. [문서 허브](./README.md)에서 현재 기준 문서를 확인한다.
2. 변경 책임에 해당하는 상세 문서만 수정한다.
3. 상태·다음 작업이 바뀌면 상위 문서와 체크리스트도 함께 수정한다.
4. 새 상세 문서를 만들면 상위 인덱스에 링크한다.
5. 줄 수, 상대 링크, Markdown 코드 펜스를 검사한다.

## 검증 명령

```bash
find docs -name '*.md' -print0 | xargs -0 wc -l | sort -nr
```

출력에서 개별 파일이 200줄을 넘으면 안 된다. 상대 링크는 저장소 루트에서 링크 검사기로 확인하고, 각 파일의 코드 펜스 수가 짝수인지 확인한다.

## 금지 사항

- 활성 계획을 과거 설계에 맞추기 위해 조용히 되돌리지 않는다.
- 상위 문서에 모든 상세 내용을 다시 복사하지 않는다.
- 링크 없이 상세 문서를 고립시키지 않는다.
- 구현되지 않은 항목을 완료로 표시하지 않는다.
- mock 연동 성공을 실제 MedGemma 연동 성공으로 기록하지 않는다.
